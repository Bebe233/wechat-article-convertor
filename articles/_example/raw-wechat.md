# 写公众号太麻烦？用 Markdown 转微信工作流把排版一次搞定

你是不是也遇到过：技术文章在本地用 Markdown 写得很爽，一到公众号后台就要重新调字号、行距、引用样式，半小时全耗在「排版」上？**Markdown转微信**（本仓库示例）想解决的就是这件事——把「写」和「贴」拆开，中间用可复用的规范衔接。

## 痛点：复制粘贴不是万能

- 从网页或笔记复制进公众号，样式经常丢光或乱码
- 每篇文章重复调 H2、引用、列表，无法沉淀团队规范
- 配图只能事后补，写文章时想不到哪里需要插图

## 方案：四阶段流水线 + 两个确认卡点

先用 Cursor 调研你的项目或网站，生成**事实表** `facts.md`，你确认后再写中文营销稿 `raw-wechat.md`；确认成稿后，才按统一 snippet 生成 `wechat.html`。这样改事实不会牵动排版，改语气也不会误触 HTML。

【highlight】核心原则：排版只允许选 `snippets/` 里的片段填内容，禁止临时发明新样式——公众号里才能长期看起来「像同一套品牌」。

## 功能亮点

- **事实表先行**：功能、差异化、不宜夸大项写清楚，避免营销瞎编
- **中文 raw 专稿**：成稿阶段不混微信 HTML，改稿更轻松
- **snippet 排版**：H2/H3/引用/列表/分割线/高亮/CTA 与设计 token 一致
- **配图占位**：`<!-- IMAGE: 描述 | 比例 | 备注 -->` 写进稿子里，排版时原样带进 HTML
- **纯 Cursor**：v1 不绑 Next.js CLI，Skill 即文档即流程

## 场景：技术团队发版笔记

假设你刚开源了一个小工具：在 Agent 里说一句「用 wechat-article 工作流，根据本仓库写一篇公众号」，提供受众和语气。Agent 会先给你 `facts.md`，你核对功能有没有写错；满意后再出 `raw-wechat.md` 改标题和故事线；最后才得到可粘贴的 `wechat.html`。

---

## 总结

Markdown 转微信不是再做一个编辑器，而是把**调研、撰稿、排版**拆成可复用步骤，让公众号发布变成「确认两次、复制一次」。本仓库 `articles/_example/` 就是标准答案，照着走即可。

**在 Cursor 中运行 `/wechat-article` 或引用 wechat-article-pipeline，从你的项目路径开始试一篇。**
